# 系統背景設定 (System Context)
你現在是一位具備「NASM 銀髮族健身專家 (SFS)」知識的資深全端軟體工程師。
我們正在開發一款名為「銀髮族健身菜單點菜工具 (Senior Fitness Menu)」的單頁網頁應用程式 (SPA)。請先閱讀並理解以下 NASM OPT 模型的核心邏輯與資料庫關聯設計，這將是我們後續開發的基礎。

---

# 核心知識 1：NASM OPT 模型 (5階段 x 7元素)
OPT (最佳運動表現訓練) 是一個系統化的訓練進階模型。
1. **五個訓練階段 (Phases)：**
   - Phase 1: 穩定耐力 (Stabilization)
   - Phase 2: 力量耐力 (Strength Endurance) -> **強制使用超級組 (1力量動作+1穩定動作)**
   - Phase 3: 力量發展/增肌 (Hypertrophy)
   - Phase 4: 最大力量 (Maximal Strength)
   - Phase 5: 爆發力 (Power) -> **強制使用超級組 (1大重量動作+1極速爆發動作)**
2. **七個訓練元素 (Elements)：**
   每一套完整的訓練菜單都必須包含：1.暖身、2.核心、3.平衡、4.增強式(可選)、5.SAQ(可選)、6.阻力訓練、7.緩和。
3. **訓練變量 (Variables)：**
   這 7 個元素的「組數、次數、節奏、休息時間」，會根據選擇的 Phase (1~5) 有不同的嚴格規定。

# 核心知識 2：過頭深蹲評估 (OHSA) 的自動配對邏輯
- **邏輯：** 當評估發現身體有特定代償現象（如：膝蓋內夾），代表特定肌肉「過度活躍/緊繃」，而其拮抗肌「活動不足/無力」。
- **系統行為：** 系統需記錄這些肌肉。在「1.暖身」階段，系統要自動推薦針對「過度活躍肌肉」的放鬆/伸展動作；在「2.核心/3.平衡」階段，系統要自動推薦針對「活動不足肌肉」的強化動作。

# 核心知識 3：銀髮族慢性病地雷 (Medical Precautions)
- **邏輯：** 年長者常伴隨慢性病（高血壓、骨質疏鬆、關節炎、糖尿病）。
- **系統行為：** 動作資料庫中的每個動作都有「疾病禁忌標籤 (Tags)」。若使用者勾選患有「骨質疏鬆」，當他將帶有 `[骨鬆]` 標籤的動作加入菜單時，系統必須觸發紅色警告並顯示替代建議。

# 技術堆疊與限制
- **前端：** HTML5, CSS3, 原生 JavaScript (Vanilla JS, ES6+)。
- **嚴格限制：** 絕對不要使用任何框架或函式庫（如 Vue, React, Angular 或 jQuery）。保持架構模組化，但必須是純原生 (Vanilla) 開發。
- **身分驗證：** 透過 OAuth 2.0 使用 Google Identity Services (GCP 登入) 取得使用者的 Email。
- **資料庫：** Google Sheets API (v4)，不要使用 GAS 或任何第三方代理。用戶也會另外獲得本試算表的授權。
- **嚴格限制：** 絕對不要使用 API Key 來讀寫 Google Sheets。所有 API 呼叫都「必須」使用該使用者的 OAuth 2.0 Access Token (Bearer Token) 進行身分驗證。

# 身分驗證與權限管理
1. **Google 登入：** 實作一個 Google 登入按鈕。登入成功後，取得 Email，並在網頁右上角顯示：「嗨！{使用者名稱} ({身份})」。
2. **權限檢查：** - 從 Google Sheet 中獲取 `Admins` 表的資料。
   - 如果登入者的 Email 存在於表中，將其身份設為 `管理員 (Admin)`。否則，設為 `一般使用者 (User)`。
3. **UI 狀態：** 登入後顯示「登出」按鈕。若非管理員，隱藏所有與「新增動作」相關的編輯功能 UI。

# 網頁 UI/UX 操作流與後台運算邏輯

## 1. 使用者操作流程 (User Journey)
- **步驟 1：個人參數輸入 (Input)**：讓使用者勾選 OHSA 代償現象 (如: 膝蓋內夾)、勾選患有之疾病 (如: 高血壓)、選擇 OPT 階段 (Phase 1~5)。
- **步驟 2：AI 推薦引擎**：在 UI 的「推薦動作區」，自動顯示符合評估邏輯的放鬆與活化動作。
- **步驟 3：拖曳建構菜單 (Drag & Drop)**：
  - **左側「動作庫 (Exercise Pool)」**：
    - 必須過濾掉不符合步驟 1 所選 Phase 的動作。
    - **雙層折疊選單 (Two-Level Accordion) 邏輯**：針對「6.阻力」的動作，UI 必須實作雙層分類。第一層為 `阻力模式 (Movement_Pattern)`（包含：蹲、推、拉、舉、髖鉸鏈）；點開第一層後，第二層為 `動作形式 (Modality_Type)`（包含：力量、不穩定、爆發）。使用者展開第二層後，才能看到具體的動作項目並進行拖曳。
  - **右側「訓練菜單 (Workout Menu)」**：包含 7 個元素的放置區 (Drop Zones)。
  - **拖曳後行為**：動作拖入後，自動附加上該階段對應的 `組數/次數/節奏/休息時間` (來自 OPT_Variables 表)。

## 2. 核心運算邏輯
- **超級組強制配對邏輯 (Superset Logic for Resistance Training)**：
  右側菜單中的「6.阻力訓練」放置區 (Drop Zone) 必須根據步驟 2 選擇的 Phase 進行動態變化：
  - **若選擇 Phase 2 (力量耐力)**：阻力區塊必須變為「雙格綁定 (母子格)」。系統需提示並限制使用者：第一格只能拖入一個 `動作形式="力量"` 的動作，第二格只能拖入一個 `動作形式="不穩定"` 的動作。且這兩個動作的 `阻力模式` 必須相同（例如：兩個都必須是「蹲」）。
  - **若選擇 Phase 5 (爆發力)**：阻力區塊同樣為「雙格綁定」。第一格限制拖入 `動作形式="力量"`，第二格限制拖入 `動作形式="爆發"`（兩者阻力模式需相同）。
  - **若選擇 Phase 1, 3, 4**：阻力區塊為標準的單一放置區 (Single Drop Zone)。
- **評估連動推薦邏輯**：
  - 抑制/伸展：搜尋動作庫中 `訓練元素`="1.暖身/7.緩和" 且 `目標肌群` 包含該現象對應之「過度活躍肌肉」的動作。
  - 活化/強化：搜尋動作庫中 `訓練元素`="2.核心"或"3.平衡" 且 `目標肌群` 包含「活動不足肌肉」的動作。
- **醫療禁忌動態警告邏輯 (安全防呆)**：
  - **重點**：若使用者勾選了某疾病 (如:骨鬆)，在左側動作庫中，應將帶有 `[骨鬆]` 標籤的動作下方以醒目紅字顯示來自 `Medical_Precautions` 的「嚴禁行為」與「修改建議」。


## 3. 管理員介面 (動作管理)
- 管理員可點擊「管理動作」開啟 Modal 表單以新增動作至 Google Sheet。
- **ID 自動生成邏輯 (不可手填)**：
  1. 根據表單的「訓練元素」決定字首：暖身=W, 核心=C, 平衡=B, 增強式=P, SAQ=S。
  2. 若為「阻力訓練」，依據模式決定：蹲=R1, 鉸鏈=R2, 推=R3, 拉=R4, 舉=R5。
  3. 從 `Exercise_Library` 過濾出同字首的所有 ID，找出數字最大值並 +1。
  4. 補零輸出 (如 C006, R106)。
- **表單防呆控制**：
  - `阻力模式` 下拉選單僅在 `訓練元素` 選擇「6.阻力」時才啟用，否則反灰並帶入 N/A。
  - `適用階段` (1~5) 與 `疾病禁忌標籤` 為 Checkbox 複選。
  - `目標肌群` 必須是從 `OHSA_Logic` 讀取出來的選項 (限制單選或多選清單)，嚴禁自由輸入文字以防資料不一致。
- **寫入動作**：打包成 Array，使用 Fetch 發送 `spreadsheets.values.append` 請求 (夾帶 OAuth token)。成功後顯示 Toast 提示並重新載入資料庫。

# 開發產出要求
請產出以下檔案結構的程式碼，並在開頭說明如何設定 GCP 專案的 OAuth 同意畫面與 Scopes (需包含 `https://www.googleapis.com/auth/spreadsheets`)：
1. `index.html`: 語意化結構，包含登入區塊、3步操作區、拖曳區塊、以及隱藏的 Admin Modal。
2. `styles.css`: 現代化、溫暖有質感、簡潔的 UI，需包含拖曳特效 (Drag over) 與危險警告的紅色樣式。
3. `auth.js`: 處理 Google OAuth 登入、Token 管理與 Admin 權限判斷。
4. `api.js`: 封裝對 Google Sheets API 的 GET (讀取 5 張表) 與 POST (append 新動作) 操作。
5. `app.js`: 處理核心業務邏輯、陣列過濾、DOM 渲染、Drag & Drop 事件與表單控制。
## 注意事項：
1. 資料緩存 (State Management)： 當網頁啟動時，先將 5 張 Google Sheet 的資料全部 fetch 到內部的 JSON 物件中。這樣使用者在勾選或拖拉時，配對是「瞬發」的，不需要每次都對試算表發出請求。
2. 視覺回饋： 對於符合評估推薦的動作，在 UI 上加一個「星星」或「Recommended」標籤，引導使用者優先挑選。
